# L1-SyncE

SyncE，同步以太网。它定义了如何从接收到的比特流（Bitstream）中提取时钟频率，以及如何控制发出的物理信号频率。

SyncE 与传统的网络协议（如 TCP/IP 或以太网帧）有本质区别：它不依赖于数据包的内容，而是直接利用物理链路上的电信号或光信号波形。

应用场景：这种设计常见于 5G 基站前传、核心交换机或电信级路由器。因为在这些场景下，仅仅有数据传输是不够的，所有设备必须在频率甚至相位上保持高度同步，否则会出现丢包、通话断续或无线信号干扰。


## 工作原理

交换芯片 BCMxxx 连接 时序板 TimingCard ZL30xxx。ZL30xxx 提供两路 DPLL0 和 DPLL1 25MHz 的恢复时钟。

> 通常需要配置交换芯片SDK中对应端口的time_synce_clock_source_control，包括主备恢复时钟，即上述中的两路。

1. 信号提取：BCMxxx 交换芯片
   - 动作：BCMxxx（博通的高性能交换芯片）从指定的 物理端口 中提取时钟信号 （通常需要配置端口的time_synce_clock_source_control）。
   - 产物：它生成了两路 25MHz 的恢复时钟（Recovered Clocks）。
   - 原理：在同步以太网（SyncE）中，时钟信号是直接嵌入在数据流里的。芯片的任务就是把这个隐含的“心跳”找出来。

2. 信号传输：发送至时钟板
   - 这两路恢复出来的 25MHz 时钟信号被送到名为 ZL30xxx 的芯片上。
   - ZL30xxx 是一颗专门的时钟发生器/同步器芯片（通常被称为网络同步时钟倍频器或锁相环芯片）。

3. 信号锁定与净化：DPLL 过程
   - DPLL0 与 DPLL1：ZL30xxx 内部有两个数字锁相环（Digital Phase-Locked Loops）。
   - 锁定（Lock）：这两个 DPLL 分别锁定到那两路 25MHz 时钟上。
   - 目的：
     - 去抖动（Jitter Attenuation）：从光口提取的时钟通常比较“脏”，有很多干扰，DPLL 可以把它变得非常稳定、纯净。
     - 频率转换：将这 25MHz 的参考信号转换成系统其他组件（如 CPU、FPGA）所需的各种不同频率。


### 以下是 SyncE 工作机制的详细拆解：

------

#### 1. 核心原理：物理层时钟恢复

传统的以太网是“异步”的，而 SyncE 把它变成了“同步”的。

- **时钟嵌入数据**：在发送端，交换芯片将高精度的时钟信号注入到物理层的比特流中。
- **时钟恢复（Clock Recovery）**：接收端（比如你的 BCMxxx）并不只是读数据，它还通过锁相环（PLL）电路，从接收到的信号波形中把那个“节奏”提取出来。这就是你提到的 **"Recovered 25M clocks"**。

------

#### 2. 为什么需要 ZL30xxx（DPLL）这个“中间商”？

既然交换机已经提取出了 25MHz 的时钟，直接用不就行了吗？为什么要送给 DPLL 芯片？

这是因为从光口提取出来的原始时钟非常“脏”：

1. **抖动（Jitter）**：由于线路干扰、光模块发热等原因，提取出的 25MHz 信号频率会忽快忽慢。
2. **漂移（Wander）**：长时间看，频率可能会偏离标准值。
3. **ZL30xxx 的作用**：它像是一个**高级滤水器**。它接收不纯净的 25MHz，利用内部的数字算法（Digital PLL）滤掉抖动，输出一个极其稳定、纯净的频率再还给系统使用。

------

#### 3. 同步以太网的“链式结构”

SyncE 的强大之处在于它可以像接力棒一样传递：

- **一级参考源 (PRC)**：通常是一个原子钟（如 GPS 授时）。
- **传递**：第一台交换机锁定原子钟，通过网线发给第二台。
- **跟踪**：第二台交换机锁定第一台的频率，再发给第三台。
- **结果**：整个城市的所有基站、交换机，心跳频率完全一致。

------

#### 4. SyncE vs. 网络校时 (NTP/PTP)

很多人会把 SyncE 和我们电脑上常用的 NTP 时间同步搞混，它们其实是两个层面的东西：

| **特性**     | **SyncE (同步以太网)**                  | **NTP / PTP (1588v2)**                           |
| ------------ | --------------------------------------- | ------------------------------------------------ |
| **同步内容** | **频率 (Frequency)**                    | **相位/时刻 (Time/Phase)**                       |
| **类比**     | 大家的节拍器跳得一样快。                | 大家的表显示的几点几分一样。                     |
| **实现层级** | **物理层 (L1)**：直接看物理信号的波动。 | **协议层 (L2/L3)**：通过收发带有时间戳的数据包。 |
| **稳定性**   | 极高，受网络拥塞影响为 0。              | 会受网络拥塞和延迟抖动影响。                     |

> **硬核知识点：** 在 5G 架构中，通常用 **SyncE 负责频率同步**（保证不掉线、不丢包），用 **PTP 负责时间同步**（保证基站之间切换不间断）。

------

#### 5. 硬件场景

1. **输入**：从 SFP 光口进来的数据流里带着远端的“标准节拍”。
2. **提取**：BCMxxx 像听诊器一样把 25MHz 的节拍抓出来。
3. **洗涤**：ZL30xxx (DPLL) 把这个节拍洗干净，消除抖动。
4. **分发**：洗干净的节拍送回给交换机和主控系统。
5. **输出**：交换机用这个洗干净的节拍去发送数据给下游设备。



### 深度思考

交换机有几十个端口、每秒处理数亿个数据包（Packet），为什么只需要“两路”恢复时钟就够了？

这背后的设计哲学是：“全班同学只需要听一个老师的节拍器，就能跳出整齐的舞步。”

#### 1. “频率”与“内容”的解耦

交换机处理数据包（Packet）和维持时钟频率（SyncE）是两个完全不同的层级：

- **数据包处理（异步）**：每个端口进来的数据包确实是独立的、海量的，它们被放进芯片内部的缓存（Buffer）里排队处理。
- **时钟频率（同步）**：SyncE 关注的是**物理层（L1）**的比特流速度。
- **核心逻辑**：整个交换芯片内部只需要**一个**核心参考时钟。只要这个核心时钟是准的，芯片发往所有端口的数据流频率都会以这个准时钟为基准。因此，不需要为每个端口都搞一套同步电路。

#### 2. 为什么是“两路”？（1+1 备份机制）

之所以选择两路恢复时钟（Recovered Clocks），主要是为了**冗余和选优**，而不是因为端口多：

- **主备切换（Redundancy）**：通常系统会从两个不同的上游端口（比如 Port A 和 Port B）分别提取时钟。如果 Port A 的光纤断了，ZL30732B 会瞬间切换到第二路时钟，保证系统同步不中断。
- **时钟选优**：系统会通过 SSM 协议比较这两路时钟的质量（比如哪路更接近原子钟），择优锁定。

#### 3. 25MHz 怎么驱动这么多的高速端口？

你可能会想：端口是 25G 甚至 100G 的，25MHz 这么慢怎么够用？

- **倍频器（Frequency Multiplier）的作用**：

  ZL30732B 锁定了这路 25MHz 的“精准节拍”后，会利用内部的合成器把它倍频成各种高速频率（比如 156.25MHz 或更高的频率）。

- **分发树结构**：

  这个高精度的频率信号会像树根一样，分发出无数个支脉，送往交换芯片内部的每一个 **SerDes（串行器/解串器）**。无论芯片有 32 个端口还是 128 个端口，它们最终都参考同一个 DPLL 输出的“指挥棒”。


#### 4. 参考源越少，系统越稳定

- 如果给每个端口都搞一套独立的同步时钟，反而会导致芯片内部不同区域的频率微小偏移，造成数据跨时钟域传输时出现错误（Meta-stability）。
- **两路 25M 恢复时钟**已经足以让这台交换机在全球同步网中找到自己的位置，并带领所有端口步调一致地工作。
